# V1.1 Supported-Language Compatibility Gate

## Goal

Verify that every target language advertised by the app works through the
complete technical workflow. Passing this gate means the app preserves,
imports, validates, and exports text correctly. It does not certify linguistic
quality or native-speaker accuracy.

## Supported Matrix

| Code | Language   | Primary compatibility concern                 |
| ---- | ---------- | --------------------------------------------- |
| `de` | German     | Existing language-specific LLM instructions   |
| `es` | Spanish    | Accented Latin characters                     |
| `fr` | French     | Accents, apostrophes, and spacing             |
| `hu` | Hungarian  | Double-accent characters such as `ő` and `ű`  |
| `it` | Italian    | Apostrophes and accented Latin characters     |
| `ja` | Japanese   | Non-Latin text without whitespace assumptions |
| `ko` | Korean     | Hangul UTF-8 round trips                      |
| `pt` | Portuguese | `pt.json` and SMAPI `pt-BR.json` fallback     |
| `ru` | Russian    | Cyrillic UTF-8 round trips                    |
| `tr` | Turkish    | Dotted and dotless `i` characters             |
| `zh` | Chinese    | Han text without whitespace assumptions       |

## Automated Verification

Use parameterized tests over all 11 target codes to verify:

1. The language can be selected and persisted in settings.
2. Scanning recognizes the expected target file for the selected language.
3. Existing translations import without character loss.
4. Editing, state persistence, rescan, and reload preserve Unicode exactly.
5. Export writes valid UTF-8 JSON with the correct target filename and key
   order.
6. Protected-token validation behaves identically for Latin and non-Latin
   translations.
7. External LLM batches contain the correct target code, language label,
   instructions, sections, and Unicode content.
8. Local-AI prompts contain the correct target language and preserve protected
   tokens.
9. Portuguese accepts the documented SMAPI fallback inputs while exporting the
   app's canonical target filename.

Tests must include representative accented Latin, Cyrillic, Hangul, Japanese,
and Chinese strings. Fixtures must be minimal synthetic content and must not
contain Stardew Valley game assets or third-party mod data.

### Automated Result (2026-06-12)

The parameterized backend compatibility test passes for all 11 target codes. It
covers settings persistence, scan/import, exact Unicode round trips, saved
edits, rescan/reload, canonical export filenames and key order, protected
tokens, external LLM batch export/import, section context, and local-AI prompt
construction. Frontend tests additionally verify that every language is listed,
selectable, and saved from Settings.

Portuguese compatibility is implemented as an import-only fallback:
`pt-BR.json` is read when `pt.json` is absent, while export always writes the
canonical `pt.json`. Scanning never renames or modifies either file.

Saved working state is isolated under `Data/language-state/<code>/`, so
switching target languages in one portable folder cannot leak edits or review
statuses between languages. Existing `Data/translations/` state is migrated
once into the language active during the upgrade.

## Manual Smoke Test

Create the disposable synthetic fixture from the repository root:

```powershell
.\scripts\create-language-smoke-fixture.ps1
```

Use the printed `Stardew path` and `Mods path` in setup. The fixture is written
under ignored `target/language-smoke/` and contains no game or third-party mod
content. Its `LLM Results` folder contains one ready-to-import result per
language, so no external service or manual JSON editing is needed.

Run the packaged-app smoke test for the representative release set:

- `de`: Latin-script baseline and existing German-specific prompt rules.
- `ja`: non-Latin Unicode and no-whitespace assumptions.
- `pt`: `pt-BR.json` import fallback and canonical `pt.json` export.

For each representative target language:

1. Select the language in Settings and restart the portable app.
2. Scan a controlled synthetic mod fixture.
3. Import one existing translation containing representative Unicode.
4. Edit and save one string with a protected token.
5. Export the mod and reopen the generated JSON.
6. Export and re-import a small external LLM batch.
7. Record the app version, date, tester, and result.

The release checklist distinguishes the complete automated matrix from the
representative packaged-app smoke coverage. Every language is technically
verified by automation; the representative set additionally verifies the real
portable Windows workflow across the main compatibility classes.

## Acceptance Criteria

- All 11 target codes pass the automated matrix.
- The representative `de`, `ja`, and `pt` set passes the packaged portable
  Windows smoke workflow.
- No test loses, normalizes, or corrupts Unicode text.
- Target filenames and Portuguese fallback behavior match SPEC Appendix A.
- Failures identify the language and workflow stage.
- Documentation says `technically verified`, not `translation quality
verified`.
